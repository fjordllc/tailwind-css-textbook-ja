# 図版指示書

本書（Tailwind CSS 教科書）の図版を作成するための指示書です。初稿（本文）は完成済みで、図版は**本文の理解を補強する別タスク**として扱います。まず「全体方針」を読み、次に「優先図版3点」を詳細仕様どおりに作成してください。残りは「追加候補」として優先度を下げています。

対象本文: `manuscript/` 配下。バージョン基準: Tailwind CSS v4 / 2026-06。

---

## 1. 全体方針（すべての図版に共通）

### トーン・スタイル
- **教科書らしい、落ち着いた図**。装飾過多にしない。情報の流れが一目で分かることを最優先。
- 背景は白。要素は角丸の矩形（`rounded` 程度）＋細い境界線。影は控えめ（本書 §13.6 の「影を盛りすぎない」と一貫させる）。
- 差し色は Tailwind のブランド系（`sky`/`blue`、必要なら強調に `amber`）を**1〜2色まで**。多色にしない。

### アクセシビリティ（本書 §21.5 と一貫させる — 重要）
- **色だけで情報を区別しない**。「良い例＝緑／悪い例＝赤」のような色分けをする場合も、必ず**ラベル文言（○ / ✕、「推奨」「非推奨」）やアイコン・パターン**を併用する。
- **モノクロ印刷でも判別可能**にする（書籍化を想定）。濃淡・破線/実線・ラベルで区別。
- 文字と背景の**コントラスト比は WCAG AA（通常テキスト 4.5:1 以上）**を満たす。
- 各図に**代替テキスト（alt）**を必ず付ける（本指示書に案を記載）。

### 体裁・フォーマット
- **ベクター形式（SVG 推奨、入稿用に PDF も）**。解像度非依存にする。
- 図中の文字は**日本語ゴシック**（本文と調和するもの）。コード片は等幅フォント。
- 図版サイズは**本文幅いっぱい（横長）を基準**。込み入る図は縦に伸ばす。スマホ/電子書籍のリフローも想定し、横 3 分割以上を1行に詰め込みすぎない。
- コード片を図に入れるときは、本文のコード例と**文言を完全一致**させる（差異は混乱の元）。

### 番号・キャプション・配置
- 図番号は **「図 章-連番」**（例: 図 4-1）。
- キャプションは図の**下**に置く。「何を表す図か」を1文で。
- 本文中に「（図 4-1）」と参照を入れる箇所も本指示書で指定する。

## 2. 優先図版3点（必須・この順で作成）

### 図 4-1 「テンプレート走査 → 必要なクラスだけ生成」
- **対応本文**: 第4章 §4.1〜4.2（`manuscript/part2/chapter4.md`）。本文参照を §4.1 の本文末に「（図 4-1）」として挿入。
- **目的（なぜ必要か）**: 本書で最も重要な仕組み「使われているクラスだけが CSS になる」を、文章より一目で理解させる。JIT の核。
- **種類**: 左→右のフロー図（3段）。
- **構成（ワイヤー）**:

```
[ソースファイル群]            [Tailwind エンジン]          [生成された CSS]
 ┌──────────────┐                                      ┌──────────────┐
 │ index.html   │                                      │ .p-6 {       │
 │  class="p-6  │   ──走査(スキャン)──▶  ⚙  ──生成──▶  │   padding:   │
 │  text-center"│      使われた文字列だけ拾う           │   calc(...)} │
 │ Card.jsx     │                                      │ .text-center │
 │  ...p-6...   │                                      │  {...}       │
 └──────────────┘                                      └──────────────┘
   ※ 登場しないクラス（例: m-99）は                      ※ 使ったクラスだけ。
     最初から生成されない                                  だから小さい
```

- **強調ポイント**: (1) ソース内の `p-6` `text-center` をハイライト（同じ色で「これが拾われる」）。(2) エンジンを通って「使ったものだけ」が右へ。(3) 右下に「使わないクラスは生成されない＝CSS が線形に増えない」の注記。
- **ラベル文言（日本語）**: 「①ソースを走査」「②使われたクラス名を収集」「③必要な分だけ CSS を生成」。
- **キャプション案**: 「図 4-1　Tailwind はソースを走査し、実際に使われているクラスだけを CSS に生成する（JIT）。」
- **代替テキスト案**: 「ソースファイルに書かれた p-6 や text-center などのクラスを Tailwind エンジンが走査し、実際に使われたクラスだけを CSS として生成する流れを示した図。使われていないクラスは生成されない。」
- **注意**: 生成 CSS は本文 §4.9 に合わせ `padding: calc(var(--spacing) * 6)` とする（`1.5rem` にしない）。

### 図 5-1 「`@theme` の変数は CSS 変数とユーティリティの両方になる」
- **対応本文**: 第5章 §5.3（`manuscript/part2/chapter5.md`）。§5.3 の生成例の近くに「（図 5-1）」を挿入。
- **目的**: v4 テーマの肝「1つの定義が、CSS 変数生成とユーティリティ生成を**同時に**担う」を視覚化する。
- **種類**: 1→2 に分岐する図（上に定義、下に2つの出力）。
- **構成（ワイヤー）**:

```
            ┌────────────────────────────────────┐
            │ @theme {                           │
            │   --color-mint-500: oklch(...);    │   ← 1つの定義
            │ }                                  │
            └───────────────┬────────────────────┘
                            │ 2つの役割に分岐
            ┌───────────────┴───────────────┐
            ▼                               ▼
 ┌──────────────────────┐      ┌────────────────────────────┐
 │ ① CSS 変数として公開   │      │ ② ユーティリティを生成        │
 │ :root {              │      │ .bg-mint-500 { ... }        │
 │  --color-mint-500:.. │      │ .text-mint-500 { ... }      │
 │ }                    │      │ → HTML で class として使える │
 │ → var(...) で使える   │      │                            │
 └──────────────────────┘      └────────────────────────────┘
```

- **強調ポイント**: 上の1ブロックから下の2ブロックへ矢印が分岐。「**単なる CSS 変数ではない**」という本文の主張を、分岐そのもので表現。
- **ラベル文言**: 「1つの定義」「2つの役割」「①実行時に使える CSS 変数」「②class として使えるユーティリティ」。
- **キャプション案**: 「図 5-1　`@theme` に書いた1つの変数が、CSS 変数とユーティリティクラスの両方を生み出す。」
- **代替テキスト案**: 「@theme で定義した --color-mint-500 という1つの変数が、:root の CSS 変数と、bg-mint-500・text-mint-500 というユーティリティクラスの両方に展開される様子を示した分岐図。」

### 図 22-1 「`@apply` vs コンポーネント抽出」
- **対応本文**: 第22章 §22.2〜22.3（`manuscript/part6/chapter22.md`）。§22.3 末に「（図 22-1）」を挿入。
- **目的**: 重複の解消法として `@apply` を選ぶと BEM 時代の問題（命名・CSS 肥大・往復）に戻るのに対し、コンポーネント抽出は利点を保てる、という対比を一目で示す。
- **種類**: 中央に「重複した長いクラス列」を置き、左右2経路へ分かれる対比図。
- **構成（ワイヤー）**:

```
                 ┌───────────────────────────────┐
                 │ 重複: 同じ長いクラス列が3か所  │
                 └───────────────┬───────────────┘
            ┌────────────────────┴────────────────────┐
            ▼                                          ▼
 ✕ 非推奨: @apply で CSS に逃がす          ○ 推奨: コンポーネントに抽出
 ┌───────────────────────────┐           ┌───────────────────────────┐
 │ .btn-primary {            │           │ <Button> / _button.erb     │
 │   @apply ...;             │           │  長いクラス列は定義の1か所  │
 │ }                         │           │                            │
 │ ・命名コストが復活         │           │ ・命名は最小限              │
 │ ・CSS が再び増える         │           │ ・CSS は増えない            │
 │ ・HTML↔CSS の往復が戻る    │           │ ・props でバリエーション可  │
 │ → BEM 時代へ逆戻り         │           │ → Tailwind の利点を保つ     │
 └───────────────────────────┘           └───────────────────────────┘
```

- **強調ポイント**: 左右で「結果」が対照的であること。左は ✕＋「逆戻り」、右は ○＋「利点を保つ」。**色だけでなく ✕/○ 記号と文言で必ず区別**（モノクロ可）。
- **ラベル文言**: 「非推奨」「推奨」、各経路の箇条書きは本文 §22.2/§22.3 の文言に揃える。
- **キャプション案**: 「図 22-1　重複への2つの対処。`@apply` は BEM 時代の問題に逆戻りし、コンポーネント抽出は Tailwind の利点を保つ。」
- **代替テキスト案**: 「重複した長いクラス列の対処を2経路で比較した図。@apply で CSS クラスにすると命名コストや CSS 肥大が復活し BEM 時代に逆戻りする。一方コンポーネント抽出は長いクラス列を1か所に閉じ込め、CSS を増やさず利点を保つ。」

#### 図 10-1 「余白の3種: padding / margin / gap」
- **対応本文**: 第10章（§10.2〜10.4、`manuscript/part4/chapter10.md`）。
- **目的（なぜ読みやすくなるか）**: 初学者が最も混同する「内側(padding)・外側(margin)・要素間(gap)」を、1枚で空間的に区別できる。
- **種類**: ボックスモデル風の構成図。
- **構成（ワイヤー）**:

```
   ┌─ margin（外側の余白：要素の外）──────────────┐
   │  ┌─ border ───────────────────────────────┐  │
   │  │  ┌─ padding（内側の余白：枠と中身の間）┐ │  │
   │  │  │        コンテンツ                  │ │  │
   │  │  └────────────────────────────────────┘ │  │
   │  └──────────────────────────────────────────┘  │
   └──────────────────────────────────────────────┘

   gap（子要素どうしの間。Flex/Grid コンテナに指定）
   ┌───────┐ ←gap→ ┌───────┐ ←gap→ ┌───────┐
   │ 子A   │       │ 子B   │       │ 子C   │
   └───────┘       └───────┘       └───────┘
```

- **ラベル文言**: 「padding＝内側」「margin＝外側」「gap＝子どうしの間」。各に対応するクラス例（`p-4` / `m-4` / `gap-4`）を添える。
- **キャプション案**: 「図 10-1　padding は内側、margin は外側、gap は子要素どうしの間。役割で使い分ける。」
- **代替テキスト案**: 「ボックスモデル上で、padding が枠とコンテンツの間（内側）、margin が要素の外側、gap が並んだ子要素どうしの間隔であることを示した図。」

#### 図 15-1 「Flexbox の2つの軸: 主軸(justify) と 交差軸(items)」
- **対応本文**: 第15章（§15.3、`manuscript/part4/chapter15.md`）。
- **目的**: 初学者最大の難所「`justify-*` と `items-*` はどっちの方向か」を、軸の向きで一目で示す。
- **種類**: 軸を矢印で示した方向図（`flex-row` と `flex-col` の2パターン併記）。
- **構成（ワイヤー）**:

```
 flex-row（横並び）                 flex-col（縦並び）
 主軸 = 横（justify-*）             主軸 = 縦（justify-*）
 ──────────────▶ justify           │ justify
 ┌──┐ ┌──┐ ┌──┐                    ▼ ┌──────┐
 │  │ │  │ │  │  ▲ items（交差軸）    │  A   │  ──▶ items（交差軸=横）
 └──┘ └──┘ └──┘  │ =縦              ┌──────┐
                                    │  B   │
                                    └──────┘
```

- **強調ポイント**: 「**主軸＝justify-*／交差軸＝items-***」を太字で。`flex-row` では主軸が横、`flex-col` では主軸が縦に変わることを2パターンで見せる（ここが特に誤解されやすい）。
- **ラベル文言**: 「主軸（justify-content）」「交差軸（align-items）」。
- **キャプション案**: 「図 15-1　`justify-*` は主軸、`items-*` は交差軸に効く。主軸の向きは `flex-row`／`flex-col` で変わる。」
- **代替テキスト案**: 「Flexbox で justify-* が主軸方向、items-* が交差軸方向の配置を制御することを示す図。flex-row では主軸が横、flex-col では主軸が縦になる。」

#### 図 17-2 「レスポンシブの上書きの重なり方（モバイルファースト）」
- **対応本文**: 第17章（§17.1、`manuscript/part5/chapter17.md`）。
- **目的**: 「無印が土台、`md:` 以上は“その幅以上”で上書き」という上書きの**重なり**を、画面幅の数直線上でどのスタイルが効くかとして見せる。`md:` を「md のときだけ」と誤解する典型を防ぐ。
- **種類**: 画面幅を横軸にした帯（範囲）図。
- **構成（ワイヤー）**:

```
 例: class="text-sm md:text-base lg:text-xl"

 画面幅 →   0        640px(sm)     768px(md)        1024px(lg) ───▶
            │            │            │                 │
 text-sm    ████████████████████████████████████████████████  ← 無印は全幅で効く(土台)
 md:text-base                        ██████████████████████████  ← md 以上で上書き
 lg:text-xl                                            ██████████  ← lg 以上で上書き
            └ 実際に効くのは「その幅で一番右まで届いている指定」＝後勝ち
 結果:      [ text-sm ]──[  text-base  ]──[   text-xl   ]
```

- **強調ポイント**: 各指定が「**その幅以上ずっと**」効くこと（点ではなく範囲）。重なった領域では右側（より大きいブレークポイント）が勝つこと。下段の「結果」帯で、幅ごとに最終的にどれが適用されるかを示す。
- **ラベル文言**: 「無印＝全画面の土台」「`md:`＝768px 以上で上書き」「重なりは“大きい方”が勝つ」。
- **キャプション案**: 「図 17-2　モバイルファースト。無印が土台で、`md:`・`lg:` は“その幅以上”を上書きする。」
- **代替テキスト案**: 「画面幅を横軸に、text-sm が全幅、md:text-base が768px以上、lg:text-xl が1024px以上で効くことを帯で示した図。広い画面ほど大きいブレークポイントの指定が上書きする。」

#### 図 19-1 「transition: 状態変化を時間でつなぐ」
- **対応本文**: 第19章（§19.1、`manuscript/part5/chapter19.md`）。
- **目的**: 「`transition` がないと一瞬で切り替わる／あると `duration` の時間をかけて滑らかに変わる」を、時間軸で対比する。`duration`・`ease` の役割が一目で分かる。
- **種類**: 時間を横軸にした before/after 比較図。
- **構成（ワイヤー）**:

```
 hover で bg-blue-500 → bg-blue-700 に変える場合

 transition なし:    ■(青)│■■■(濃青)        ← 0ms で瞬間的に切替（カクッ）
                          ▲ hover した瞬間

 transition あり:    ■(青)→▦→▦→■(濃青)      ← duration-200 の 200ms かけて滑らかに
 (transition         │────────────────▶ 時間
  duration-200)      0ms      ease(緩急)    200ms
```

- **強調ポイント**: 上=瞬間切替、下=時間をかけた補間。`duration-*`＝かける時間、`ease-*`＝緩急（速度カーブ）であることをラベルで。
- **ラベル文言**: 「`transition` なし＝瞬間」「`transition` あり＝`duration` の時間で補間」「`ease-*`＝速度の緩急」。
- **キャプション案**: 「図 19-1　`transition` は状態の変化を `duration` の時間で滑らかにつなぐ。`ease-*` は変化の緩急を決める。」
- **代替テキスト案**: 「ホバーで背景色が変わるとき、transition なしでは瞬間的に、transition あり（duration-200）では200ミリ秒かけて滑らかに変化することを時間軸で対比した図。」

- **着手順**: 図 4-1 → 図 5-1 → 図 22-1（優先3点）→ **3.1 の読みやすさ重視の図（まず 図 10-1・図 15-1、次に 図 17-2・図 19-1）** → 3.2 のその他候補。
- **レビュー**: 各図のドラフト段階で、(1) 本文との文言一致、(2) 色だけに依存していないか、(3) alt テキストの妥当性、を確認する。
- **納品物**: 図ごとに SVG（編集可）＋ PDF（入稿用）＋ 本指示書の alt テキスト。ファイル名は `fig-4-1.svg` のように図番号に揃える。
- **本文への反映**: 図の確定後、本文の該当位置に「（図 N-M）」の参照とキャプションを差し込む（本作業は別コミットで実施）。
